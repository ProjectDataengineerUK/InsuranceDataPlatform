## Summary

Construímos o MVP de código de uma plataforma de dados de seguros 100% Databricks: ingestão de dados públicos reais (SUSEP, ANS) via Kafka (Confluent Cloud), processamento em Spark Structured Streaming puro (sem DLT/Lakeflow) em arquitetura medallion (Bronze/Silver/Gold), detecção de fraude em streaming, aprovação automática, monitoramento de SLA, treino de modelo via MLflow, governança via Unity Catalog e CI/CD via GitHub Actions + Databricks Asset Bundles + Terraform. O projeto nasceu de uma conversa de ideação externa (ChatGPT, capturada em `context.md`) e foi levado através do fluxo completo brainstorm → define → design → build. O valor de negócio é duplo: um projeto de portfólio pronto para demonstrar competência end-to-end em engenharia de dados moderna, com uma arquitetura desenhada para evoluir para um sistema de produção real sem retrabalho estrutural.

## What Was Built

### Components

| Component | Description |
|-----------|-------------|
| Kafka Producer | Replica eventos dos datasets públicos (SUSEP/ANS) para o Confluent Cloud, em ritmo configurável |
| Bronze Ingestion Job | Spark Structured Streaming — Kafka → Delta raw, com quarentena explícita de eventos malformados |
| Silver Transformation Job | Dedup, normalização, checks de qualidade (not_null, unique) via `MERGE` upsert |
| Data Quality Module | Framework custom em PySpark (not_null, unique, range, freshness) — substitui `@dlt.expect` |
| Gold Aggregation Job | Agregações de negócio + score de fraude + regra de aprovação automática |
| Fraud Scoring Module | Heurística de 3 features (evento noturno, frequência, outlier de valor) em streaming |
| ML Training Pipeline | Treino de classificador (scikit-learn) com MLflow, usando o score heurístico como rótulo fraco |
| SLA Monitoring | Alerta (webhook) para sinistros pendentes além do SLA configurado |
| Unity Catalog (Terraform) | Catálogos/schemas por ambiente (dev/staging/prod), secrets, grants |
| Databricks Asset Bundles | 4 jobs (bronze, silver, gold, ml_training) com deploy por ambiente |
| CI/CD (GitHub Actions) | Lint + testes + validação de bundle + deploy Terraform/DAB (dev → prod gated) |

## Lessons Learned

### Process

- O fluxo brainstorm → define → design → build funcionou bem para um projeto greenfield: cada fase reduziu ambiguidade antes de comprometer código. O ponto mais valioso foi o brainstorm ter capturado a mudança de escopo do usuário ("pode incluir tudo") **antes** de qualquer linha de código ser escrita.
- Question fadigue é real: por volta da 9ª/10ª pergunta no brainstorm, o usuário passou a responder de forma mais direta e menos estruturada (ex: "O 2 POREM COMPLETO"). Da próxima vez, vale agrupar perguntas relacionadas mais cedo em vez de sequenciá-las uma a uma quando o contexto já está rico (ex: quando um `context.md` extenso já existe).
- O DEFINE assumiu números (latência, volume) como "assunções a validar" em vez de bloquear a fase pedindo esses números ao usuário — isso permitiu avançar rápido, mas gerou os `❌`/`⏭️` na verificação de Success Criteria acima. Em um cenário real, valeria um spike técnico curto entre DEFINE e DESIGN para validar essas assunções antes de desenhar a arquitetura de capacity.

### Technical

- Escrever o Bronze/Silver/Gold sem DLT (decisão explícita do usuário) exigiu reimplementar manualmente coisas que o Lakeflow resolveria de graça: checkpointing, MERGE upsert, schema evolution e — o mais notável — quarentena de eventos malformados, que só foi lembrada ao verificar o AT-002 do DEFINE contra o código já escrito. Isso confirma que "sem framework declarativo" tem um custo real de superfície de bugs, não só de linhas de código.
- Dois bugs de "primeira execução" apareceram no mesmo padrão (Gold com `replaceWhere` em tabela inexistente; uso de `DeltaTable.isDeltaTable` com nome de catálogo em vez de path) — ambos só aparecem quando a tabela ainda não existe, um cenário fácil de esquecer ao programar pensando em "estado já estabilizado". Vale sempre perguntar explicitamente "o que acontece na primeira execução, quando a tabela de destino não existe?" para qualquer sink Delta.
- Terraform com backend `local` rodando em CI (GitHub Actions) é um anti-padrão sutil — funciona localmente, mas perde o state a cada execução em um runner efêmero. Só foi percebido numa segunda leitura do `deploy.yml`, não no momento de escrever `main.tf`.

### Communication

- A pergunta sobre "portfólio vs. produção real" logo no início do brainstorm foi a mais importante da sessão — moldou a decisão de "1 workspace com catálogos por ambiente" em vez de workspaces separados, que por sua vez afetou praticamente todo o resto do design (Terraform, DAB, custo).
- Quando o usuário pediu para pesquisar o Open Insurance Brasil, a pesquisa web ativa (em vez de assumir) evitou prometer uma integração que teria ficado bloqueada por credenciamento externo — a decisão de tornar essa fonte um "fast-follow" só foi possível por verificar a informação em vez de supor.

### Tools & Libraries

- `ruff` com `select = ["E", "F", "I", "UP", "B"]` pegou um bug real de correção (B904 — `raise ... from err`), não só estilo — vale manter essa regra ligada por padrão em projetos Python novos.
- Sem Java/JVM disponível no ambiente de build, a verificação ficou limitada a `py_compile` + `ruff` + revisão manual da lógica dos testes. Isso foi suficiente para pegar bugs de lógica de dados (ver Technical acima), mas não substitui execução real — importante não confundir "sintaticamente válido" com "funcionalmente correto" ao reportar status.

## Recommendations for Future Work

| Area | Recommendation |
|------|----------------|
| Gold sem quality checks | Adicionar `check_not_null`/`check_unique` explícitos em `gold_aggregate.py`, hoje só a Silver tem gate de qualidade formal |
| Fraude não é streaming < 1 min | Reavaliar se o job Gold agendado a cada 10 min atende ao negócio, ou se o score de fraude precisa rodar como job contínuo separado (mais próximo do Silver) para cumprir o Success Criteria original de "< 1 min" |
| Validação de datasets reais | Baixar os CSVs reais da SUSEP/ANS, validar `COLUMN_MAPPING` em `susep_loader.py`/`ans_loader.py`, e resolver as `dataset_url` em `config.yaml` |
| Execução real do pipeline | Provisionar Confluent Cloud, Databricks workspace e HCP Terraform; rodar o CI/CD de ponta a ponta pelo menos uma vez em `dev` para validar os 5 Success Criteria ainda não verificados |
| Open Insurance Brasil | Iniciar o processo de credenciamento no diretório sandbox como item de roadmap separado (fast-follow já documentado no DESIGN) |
| Publicação do repositório | Publicar no GitHub público, conectar os secrets do `deploy.yml` e validar o primeiro deploy real |
