Kafka

ingestão & observabilidade

O producer publica em rajadas agendadas (GitHub Actions), não always-on — e o consumidor real do lado Databricks é Spark, não um consumer group clássico. Essa segunda parte é onde a maioria dos candidatos erra.

O que é uma partição e por que ela é a unidade real de paralelismo e ordenação?

Básico

Um tópico Kafka é dividido em partições; cada partição é um log ordenado e imutável, e a ordem só é garantida dentro de uma partição, nunca entre partições do mesmo tópico. Neste projeto, os tópicos operacionais (claim-opened, policy-created, customer-updated) e o tópico regulatório único (regulatory-claim-report, compartilhado por 3 fontes fictícias) são a unidade que o Bronze consome — cada partição vira uma unidade de paralelismo de leitura no Spark, e cada uma delas é rastreada separadamente pelo kafka_lag_reporter.

O que um consumer group garante — e o que ele não garante?

Básico

Garante que, dentro de um grupo, cada partição seja consumida por só um membro por vez (balanceamento de carga) e que o progresso seja rastreável via offset commitado no tópico interno __consumer_offsets. Não garante "exactly-once" por si só — isso depende de como o consumidor trata reprocessamento em caso de falha antes do commit. É exatamente esse detalhe que separa um consumer group Kafka nativo do que o Spark Structured Streaming faz (ver seção seguinte).

Por que este projeto usa duas API Keys diferentes do Confluent Cloud?

Intermediário

CONFLUENT_API_KEY/SECRET é uma API Key de cluster (escopo Kafka: produce/consume/admin de offsets) — usada pelo producer e pelo kafka_lag_reporter. CONFLUENT_METRICS_API_KEY/SECRET é uma Cloud API Key de conta (escopo "Global", exige papel OrganizationAdmin) — a única que tem permissão pra consultar a Metrics API (consumer_lag_offsets, received_bytes). Confundir os dois escopos é o erro mais comum ao integrar observabilidade externa num pipeline Kafka — a API key de cluster simplesmente não tem escopo pra isso, e o erro que volta não deixa isso óbvio.

Por que um consumer Spark Structured Streaming não aparece como consumer group real no Kafka?

Sênior

Porque a tolerância a falhas do Structured Streaming é 100% controlada pelo seu próprio checkpoint (offsets gravados no Delta/DBFS), não pelo protocolo de consumer group do Kafka. O conector Kafka do Spark lê offsets diretamente, sem nunca commitar de volta pra __consumer_offsets. Do ponto de vista do broker, esse "consumidor" é invisível como grupo — o que quer dizer que qualquer ferramenta de terceiros que dependa de consumer group nativo (como a Confluent Metrics API pra consumer_lag_offsets) nunca vai reportar nada pra esse job, mesmo que ele esteja processando dado perfeitamente saudável.

Como dar visibilidade externa de lag pra um consumidor que não usa consumer group nativo?

Staff

A saída aqui foi um commit administrativo de offset via KIP-496 (AdminClient.alter_consumer_group_offsets), chamado a cada micro-batch do bronze_ingest.py com um group_id sintético (databricks-bronze-<topic>). Isso não muda em nada a lógica de streaming real — é só um "carimbo" administrativo pra existir um consumer group de verdade que a Metrics API consiga enxergar. O detalhe fino: isso roda num try/except amplo e nunca deve derrubar a ingestão — perder uma atualização de lag é aceitável, perder um batch de dado não é. É esse tipo de escolha assimétrica de risco que separa "funciona no demo" de "sobrevive em produção".

Spark Structured Streaming

micro-batch, checkpoint, trigger

A pergunta que separa quem só rodou um tutorial de quem operou isso em produção: "o que acontece quando o job reinicia no meio de um batch?" — a resposta certa está inteira no checkpoint, não no Kafka.

Qual é a ideia central do Structured Streaming — "stream como tabela ilimitada"?

Básico

O Structured Streaming trata um stream como uma tabela que cresce indefinidamente, e você escreve a mesma DataFrame API/SQL que escreveria contra uma tabela estática. O motor decide, por trás, como processar incrementalmente só o que é novo — isso é o que permite reaproveitar silver_transform.py (dedup + checks de qualidade) com a mesma lógica de DataFrame usada em qualquer outro lugar do projeto.

Por que checkpoint é o mecanismo real de tolerância a falhas, não o Kafka?

Intermediário

O checkpoint (em /Volumes/<catalog>/bronze/_checkpoints/<topic> neste projeto) grava os offsets já processados e o estado da query. Se o job cair e reiniciar, ele retoma exatamente de onde parou — sem depender de nenhum commit no lado do Kafka. É por isso que startingOffsets=earliest é seguro aqui: na prática, o checkpoint já existente manda mais que essa configuração assim que o job já rodou uma vez.

trigger(availableNow=True) vs processingTime vs continuous — quando usar cada um?

Intermediário

processingTime mantém o cluster ativo continuamente, microbatch a microbatch, num intervalo fixo. availableNow=True processa tudo que chegou desde o último checkpoint e termina sozinho — ideal pra rodar como job agendado, sem manter compute alocado entre execuções. O "continuous trigger" real do Spark (baixíssima latência, experimental) é outra coisa e nem entra aqui. É fácil confundir isso com "job continuous" do Databricks Jobs (que é sobre o agendador reiniciar o job, não sobre o trigger do Spark) — e essa ambiguidade de nomenclatura é armadilha clássica de entrevista.

Qual o papel de foreachBatch, e por que ele é o ponto certo pra side-effects?

Sênior

foreachBatch dá acesso à DataFrame de cada micro-batch como uma tabela estática — o que permite fazer coisas que uma streaming query "pura" não deixa (múltiplos sinks, agregações complexas, side-effects). Neste projeto, _write_batch usa isso pra: (1) separar registros válidos de quarentena, (2) escrever os dois em tabelas Delta diferentes, e (3) calcular o offset máximo por partição e reportar pro kafka_lag_reporter — três responsabilidades que não caberiam numa writeStream direta sem foreachBatch.

Um bug real: por que "CANNOT_DETERMINE_TYPE" acontece com uma coluna inteiramente nula?

Staff

measure_pipeline_latency.py monta um relatório onde, sem eventos recentes, a chave within_sla fica ausente em todos os 3 relatórios — a coluna inteira do createDataFrame fica None. O Spark clássico infere isso; o Spark Connect (usado neste compute serverless) não consegue inferir tipo de uma coluna 100% nula e falha com CANNOT_DETERMINE_TYPE. A correção foi declarar um StructType explícito pro schema do relatório — um lembrete de que Spark Connect não é um drop-in 1:1 do Spark clássico, e comportamento de inferência de schema é exatamente onde essas diferenças aparecem.

Databricks & Unity Catalog

governança, IaC, cota

Bundles (DABs) tratam jobs, apps e permissões como código — mas cota de compute serverless num workspace trial é um limite físico que nenhum YAML resolve sozinho.

O que é a hierarquia catalog.schema.table do Unity Catalog?

Básico

Catalog é o nível mais alto (aqui, um catálogo por ambiente: insurance_dev/insurance_staging/insurance_prod), schema agrupa tabelas por camada (bronze, silver, gold, monitoring), e table é a unidade final. Essa hierarquia é o que permite o mesmo código de pipeline rodar em 3 ambientes trocando só a variável catalog do bundle, sem duplicar lógica.

O que são Databricks Asset Bundles e por que usar isso em vez de clicar na UI?

Básico

DABs descrevem jobs, pipelines, apps e permissões como YAML versionado — o mesmo resources/jobs.bronze.yml é aplicado em dev/staging/prod via databricks bundle deploy -t <target>, disparado pelo CI/CD (.github/workflows/deploy.yml). Isso dá revisão de código, histórico de mudança e reprodutibilidade — o oposto de configurar um job manualmente na UI e torcer pra lembrar depois o que foi feito.

O que são "system tables" e por que system.billing.usage pode simplesmente não existir?

Intermediário

System tables são tabelas nativas do Unity Catalog com telemetria da própria conta (billing, auditoria, lineage) — mas precisam ser habilitadas explicitamente pelo account admin (Settings → Feature enablement) antes de existirem. A página de Custos deste app trata a ausência disso como um try/except que gera um aviso explicativo, nunca um erro que derruba a página — o mesmo padrão de degradação graciosa usado em qualquer tabela que só existe após um bootstrap.

Por que um Databricks App não enxerga o resto do repositório do bundle?

Sênior

Porque source_code_path em resources/visualization.yml restringe o deploy do app só a apps/insurance_platform_app — os YAMLs de jobs, Terraform e o resto de src/ nunca são sincronizados pro runtime do app. Isso importa na prática: qualquer "conhecimento de arquitetura" que o app precise mostrar (como cadência de jobs na página de Custos) precisa estar hardcoded como constante Python dentro do próprio app, porque não há como ler os YAMLs em runtime.

Compute serverless vs clássico: por que RESOURCE_EXHAUSTED acontece mesmo com jobs "leves"?

Staff

Compute serverless em workspaces trial tem cota agregada baixa — não é sobre o tamanho de cada job, é sobre quantos jobs disputam compute ao mesmo tempo. Neste projeto, mesmo com 7 jobs "leves" (a maioria só spark_python_task simples), rodar todos como continuous (24/7) simultaneamente estourava a cota — cada continuous job segura compute o tempo todo, mesmo sem dado novo pra processar. A saída não foi pedir mais cota (fora de escopo num workspace trial): foi converter os jobs de continuous pra schedule com availableNow=True, trocando "sempre ligado" por "liga, processa tudo que acumulou, desliga" — mesmo throughput eventual, fração do compute residente.

Arquitetura Medallion & Qualidade de Dados

bronze/silver/gold, DQ, SCD2

Um framework de DQ existe pra registrar, não pra bloquear — a decisão de "o que falha o pipeline vs o que só fica registrado pra auditoria" é a parte que diferencia design maduro de design ingênuo.

O que cada camada Bronze/Silver/Gold garante — e o que ela explicitamente não garante?

Básico

Bronze é o dado bruto, validado só estruturalmente (schema parseável, chaves obrigatórias presentes) — defeito de negócio (data ausente, valor inválido) passa de propósito. Silver dedup e aplica checks de qualidade de verdade (not_null, unique). Gold é o dado pronto pra consumo, com regra de negócio aplicada (score de fraude, conformidade SUSEP). Nenhuma camada garante "dado perfeito" — cada uma garante um contrato específico, e misturar as responsabilidades é o erro de design mais comum nesse padrão.

O que é quarentena/dead-letter e por que ela evita que 1 registro derrube o pipeline inteiro?

Básico

Em vez de falhar o batch inteiro quando um registro não bate o schema esperado (_is_malformed), esse registro é isolado numa tabela separada (bronze.<tabela>_quarantine), e o resto do batch segue normalmente. Essa tabela só é criada no primeiro evento malformado — um tópico saudável, sem nenhum registro quarentenado, nunca cria essa tabela, o que teve implicação direta num bug real de UI (ver Casos reais, adiante).

Quando faz sentido construir um framework de DQ próprio em vez de usar Great Expectations/dbt tests?

Intermediário

Ferramentas como Great Expectations trazem uma biblioteca rica de expectations e relatórios prontos, com custo de integração e overhead de dependência. Um framework custom em PySpark puro (check_not_null, check_unique, check_range, check_allowed_values, check_freshness) faz sentido quando o conjunto de checks é pequeno e estável, e a decisão de arquitetura já é "100% Spark, sem framework externo" — é uma troca de flexibilidade/maturidade por simplicidade e controle total do comportamento.

SCD Tipo 2 — por que is_current/valid_from/valid_to é o padrão certo pra consentimento?

Sênior

Consentimento (Open Insurance) precisa responder "quando isso mudou", não só "qual é o estado atual" — é dado de auditoria por natureza. SCD2 fecha a versão vigente antiga (is_current=false, valid_to preenchido) e abre uma nova, preservando o histórico completo. A implementação aqui trata eventos fora de ordem explicitamente: um evento mais antigo que a versão já vigente não fecha nem substitui nada — sem essa checagem, um evento atrasado da rede poderia reabrir uma versão que já deveria estar fechada.

Reconciliação entre fontes discordantes — por que mediana como resolução de conflito?

Staff

Quando 3 fontes fictícias reportam o mesmo sinistro com valores diferentes, a mediana é robusta a uma única fonte discrepante, determinística e não introduz viés de "escolher uma seguradora como fonte de verdade" — média seria puxada por outlier, e "primeira fonte que chegou" seria arbitrário e não reprodutível. É uma decisão de MVP documentada como tal, não uma alegação de que é o que um processo regulatório real faria — saber a diferença entre "decisão pragmática documentada" e "solução definitiva" é exatamente o tipo de nuance que sinaliza senioridade numa entrevista.

MLOps & Shadow Scoring

champion/challenger, drift, retrain

O modelo de fraude nunca decide sozinho neste projeto — e essa restrição não é timidez, é a resposta certa pra um rótulo de treino que vaza da própria heurística.

O que é "shadow mode" e por que ele existe antes de um cutover?

Básico

Shadow mode roda um modelo novo em paralelo ao decisor real, só pra observação — model_fraud_score é escrito em gold.claims, mas fraud_flag/auto_approved continuam decidindo 100% pela heurística de streaming_score.py. É a forma segura de comparar "modelo novo vs decisor atual" com dado de produção real, sem risco de uma decisão ruim chegar ao usuário final antes de confiança suficiente.

Por que treinar sobre rótulo derivado da própria heurística é vazamento de rótulo?

Intermediário

Se o rótulo de treino é literalmente a saída da heurística que já decide fraud_flag, o modelo aprende a reproduzir a heurística, não a superar ela — daí o f1=1.0 suspeito. Um F1 perfeito em produção quase sempre é sinal de vazamento, não de modelo excelente. É exatamente esse motivo, documentado explicitamente no código, que justifica manter o modelo em shadow mode em vez de promovê-lo a decisor: promover agora seria trocar uma heurística por uma cópia estatística da mesma heurística.

Como funciona a detecção de drift, e por que "nunca falha o pipeline sozinho" é intencional?

Intermediário

model_drift.py compara baseline_mean vs current_mean por feature (drift_ratio), rodando a cada 6h. Drift é sinal de "investigar e talvez retreinar", não uma condição binária de falha — um pipeline que trava sozinho a cada drift detectado geraria alarme falso demais pra ser operável. O pior status real que o app mostra pra drift é "atenção" (amarelo), nunca "vermelho" — decisão consciente de UX de observabilidade, não limitação técnica.

Por que retrain condicional é mais maduro que "retreinar sempre a cada 6h"?

Sênior

train_model.py só treina de fato se: é a 1ª execução (baseline ainda não existe), a última checagem de drift sinalizou algo, ou --force foi passado explicitamente. Retreinar sem necessidade queima compute (custo real) e introduz variância desnecessária no modelo em produção — cada novo champion é uma mudança de comportamento que precisa de razão pra existir, não um cron sem critério.

Por que um champion novo só é pego no próximo restart de um job contínuo — e o que isso implica?

Staff

load_champion_model_udf carrega o modelo via mlflow.pyfunc.spark_udf uma vez por run do job, não por micro-batch. Enquanto fraud_score_stream era continuous, um champion novo promovido a cada ciclo de treino só entrava em vigor no próximo restart do job — que podia demorar dias sem acontecer sozinho. A conversão pra schedule (a cada 5 min) resolveu isso de graça: cada execução agendada já é um "restart" natural, então o lag entre "novo champion promovido" e "novo champion em produção" caiu de indefinido pra no máximo 5 minutos — um efeito colateral positivo de uma decisão que foi tomada por outro motivo (cota de compute).

FinOps

DBU, cota, custo vs latência

"Lento" e "caro" no streaming quase sempre têm a mesma causa raiz: compute alocado o tempo todo, competindo por uma cota que não cresce com o número de jobs.

O que é DBU e por que compute é cobrado mesmo sem processar dado novo?

Básico

DBU (Databricks Unit) mede consumo de compute, não volume de dado processado. Um job continuous mantém um cluster/compute serverless alocado 24/7 esperando o próximo micro-batch — isso é cobrado independente de ter chegado 1 evento ou 1 milhão naquela janela. É o motivo raiz pelo qual "sempre ligado" e "caro" andam juntos em streaming mal dimensionado.

Continuous vs schedule: por que "sempre ligado" nem sempre significa "mais rápido"?

Intermediário

Quando vários jobs continuous disputam a mesma cota apertada, cada um recebe uma fatia menor e mais instável de compute — na prática, mais lento que um job schedule que ganha uma janela de compute dedicada a cada execução. Foi exatamente esse padrão que já apareceu 2 vezes neste projeto (fraud_score_stream primeiro, depois bronze_ingest/ silver_transform/regulatory_bronze_ingest): continuous parecia "mais real-time" no papel, mas na prática processava menos por estar competindo o tempo todo.

Quando reaproveitar um warehouse serverless pré-provisionado em vez de criar um novo?

Intermediário

Contas trial geralmente não permitem provisionar um segundo endpoint serverless via API (confirmado neste projeto: um databricks_sql_endpoint novo ficou preso em "Still creating..." por 15+ minutos, apply cancelado). Reaproveitar o warehouse serverless já existente ("Serverless Starter Warehouse") via output do Terraform, em vez de tentar criar um novo, evita esse problema inteiro e ainda é mais barato — um endpoint a menos consumindo cota.

Como diagnosticar "pipeline lento" como contenção de cota, e não Kafka/Spark lentos?

Sênior

O padrão de evidência que fechou esse diagnóstico aqui: (1) o producer (GitHub Actions) tinha rodado com sucesso minutos antes — ou seja, havia dado novo chegando no Kafka; (2) Bronze/Silver não avançavam havia horas, mesmo assim; (3) Gold parecia "fresco" só porque um job schedule separado (fraud_score_stream) continuava processando um backlog acumulado em Silver, não dado novo. Sinistros fora do SLA e discrepâncias de reconciliação no mesmo momento reforçaram "backlog", não "pico pontual". Nenhuma métrica isolada prova contenção de cota — é o cruzamento de sinais que prova.

Cadência escalonada entre jobs como ferramenta de FinOps — como isso funciona na prática?

Staff

Depois de converter 3 jobs de continuous pra schedule, todos rodando exatamente no mesmo minuto recriaria o mesmo pico de contenção só que periódico em vez de constante. A correção foi escalonar offsets: bronze_ingest a cada 5min (0 */5 * * * ?), silver_transform a cada 5min com +2min de offset (0 2-59/5 * * * ?, dando tempo do Bronze escrever antes da Silver ler), regulatory_bronze_ingest a cada 10min com +4min de offset. Nenhum aumento de compute — só redistribuição no tempo. É o tipo de otimização que não aparece em nenhum dashboard de custo, só na ausência de um problema que deixou de acontecer.

Casos reais de produção

o que só quem operou isso sabe contar

Cinco histórias curtas, prontas pra responder "me conta de um bug difícil que você já resolveu" sem soar genérico.

None virando NaN silenciosamente na serialização JSON regulatory_feeds.py / kafka_publisher.py

Um producer fictício usava None pra simular campo ausente. Como a lista de dicts virava uma coluna pandas mista de números e None, pd.DataFrame(rows) coagia a coluna inteira pra float64, convertendo None em NaN — e json.dumps de um float('nan') produz o token JSON inválido NaN (não null), que quebrava o cast do lado Spark com CAST_INVALID_INPUT. Corrigido na origem: sanitizar qualquer float NaN pra None logo antes de serializar, cobrindo qualquer fonte futura que caia na mesma armadilha do pandas.

SLA de latência estruturalmente inatingível numa cota compartilhada measure_pipeline_latency.py

O alvo original do DEFINE era score de fraude visível em < 1 min. Com fraud_score_stream virando schedule a cada 5 min (por contenção de cota), esse SLA reprovava sempre, estruturalmente — mesmo com o pipeline saudável. Recalibrado pra 420s (7 min), cobrindo o pior caso realista de um claim que chega logo após um ciclo começar. A lição: a métrica certa é a que o sistema pode honrar de verdade, não a que o documento original pediu antes da restrição de infraestrutura existir.

Consumer lag "sempre vazio" não é ausência de problema confluent_metrics.py / kafka_lag_reporter.py

Spark Structured Streaming não participa de consumer group nativo (ver seção Kafka) — sem o workaround deliberado de commit administrativo via KIP-496, a Confluent Metrics API mostra "nenhum consumer group ativo" mesmo com o pipeline 100% saudável. Um candidato júnior lê isso como "sem problema". Um sênior sabe perguntar "esse zero é real, ou é a ferramenta de observabilidade que não enxerga esse tipo de consumidor?" antes de tirar qualquer conclusão.

"Gold parece fresco" pode significar "processando backlog", não "pipeline saudável" status.py / fraud_score_stream

_scored_at em gold.claims reflete quando o job escreveu a linha, não a idade do evento original. Um job schedule saudável continua avançando _scored_at mesmo processando dado antigo acumulado, enquanto a ingestão real (Bronze/Silver) está parada há horas. Cruzar com sinistros fora do SLA e discrepâncias de reconciliação no mesmo momento foi o que revelou "backlog", não "tudo bem" — uma métrica de freshness sozinha, sem contexto de volume/backlog, mente por omissão.

Uma API externa tem contrato próprio, e "SQL genérico" não se aplica a ela confluent_metrics.py

A Confluent Metrics API rejeita qualquer agregação além de SUM/nulo pra essas métricas — usar MAX devolvia 400 Bad Request. Como a granularidade já era PT1M (o menor bucket suportado), SUM não soma múltiplos pontos de verdade nesse caso — é, na prática, o único valor reportado naquele minuto, não uma agregação com perda de informação. A lição: tratar o corpo da resposta de erro como fonte de verdade (a mensagem padrão de requests não inclui o corpo, onde a Confluent de fato explica o motivo) evita ter que redescobrir isso do zero na próxima vez que a API rejeitar algo.
